# duckvep correctness

Our accuracy oracle is **Ensembl VEP itself** (decades of curated
knowledge), not fastVEP. Every figure below is read **directly from a
recorded CSV** under `correctness/data/`, regenerated by
`correctness/vep_concordance.py` (offline Ensembl VEP) and the synthetic
corpus generator `tools/gen-hard-variants` (Rust + noodles).

All three annotators are compared on the **canonical normalized variant
key** (`normalize_variant`, matching VEP’s minimal representation), so
the join is valid for *every* variant class including indels.

**Method (no slop):** VEP code, VEP cache and the duckvep GFF3 are the
**same Ensembl release** (no version skew), variants are biallelic
ClinVar, and concordance is the fraction of `(variant, transcript)`
pairs whose `&`-joined SO consequence set is identical. **We never
report a single aggregate** — an average hides *where* the misses are.
fastVEP’s headline “99.99%” concealed highly significant discordances
concentrated in HIGH-impact sites. So every figure is split by **IMPACT
× variant class**, and HIGH-impact discordances are surfaced.

**Reading it** (all figures inline-generated from the CSV above):

- **SNVs are near-perfect at every impact tier** — duckvep
  `discordant/pairs` HIGH 1/43294, MODERATE 1/403438, LOW 5/214055,
  MODIFIER 6/654709 — and ahead of fastVEP (MODERATE 19/403438, LOW

- **Indels and MNVs are the residual frontier**, concentrated at HIGH

  699/1435. These are the clinically actionable sites an aggregate %

- **duckvep and fastVEP are now validly comparable on indels** —

  (HIGH del fastVEP 2187/22685 vs duckvep 2122/22685; before

  The two are nearly identical, so the indel/MNV frontier is **shared
  engine behavior** vs Ensembl VEP — a real engine gap to close, not
  duckvep-specific and not a measurement artifact.

## Synthetic hard-variant corpus

Random ClinVar sampling almost never places an indel *exactly* on a
splice landmark, so it cannot measure the splice/indel fixes.
`tools/gen-hard-variants` (a Rust binary using noodles + the parquet
cache — no R/bash/Python) constructs a corpus of pathological variants
at **exact** offsets computed from the transcript model —
deletions/insertions spanning the donor, acceptor, 5th base, donor
region, polypyrimidine tract and splice region, plus CDS
frameshift/inframe indels and in-codon MNVs — to stress-test those
classes directly.

## Reproduce

``` sh
# 1. build the Ensembl-sourced cache (server-free, from MySQL dumps)
correctness/cache-build/build-cache.sh homo_sapiens 116 38
# 2. impact-stratified concordance vs offline Ensembl VEP
correctness/vep_concordance.py data/giab/clinvar.vcf.gz \
  data/giab/GRCh38.116.gff3.gz data/giab/GRCh38.primary.fa 50000
# 3. synthetic hard-variant corpus (Rust + noodles)
cargo build -p duckvep-tools --release
./target/release/gen-hard-variants \
  data/cache/homo_sapiens.116.38/exons.parquet \
  data/cache/homo_sapiens.116.38/transcripts.parquet \
  data/giab/chr17.fa 17 correctness/data/hard_corpus.vcf
# 4. render this report
make correctness
```

See [`../PATCHES.md`](../PATCHES.md) for the accuracy patches and
[`cache-build/README.md`](cache-build/README.md) for the cache builder.
